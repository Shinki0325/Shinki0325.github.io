# 2026-07-07 首页参考站 UI 重建设计

## 概述

本设计用于把当前 Astro 资料归档站的首页与全站外壳，重建为接近 `heiehiehi/XinghuisamaBlogs` 前台的视觉与交互风格，同时保留现有站点的内容模型、GitHub Pages 静态发布方式、资料页阅读结构与后续可维护性。

本次采用“视觉等效重建”而不是直接拷贝参考站前台代码。

原因如下：

- 当前站点是 Astro 静态站，参考站前台是 Next.js App Router + React Client Components + 全局状态驱动。
- 现有站点已经承载文稿、资料页、笔记、专题、互链等内容结构，直接硬移植参考站组件会显著提高后续维护成本。
- 目标是“首页观感与交互尽量贴近参考站”，而不是复刻其技术栈。
- 为避免版权与维护风险，不直接复用参考站图片资源、文案内容和完整前端实现，仅参考其布局、交互、材质、动效与信息组织方式。

## 用户确认后的目标

- 首页整体观感尽量接近参考站。
- 顶部导航桌面端与移动端菜单尽量复刻参考站交互。
- 首页音乐功能保留参考站的“云音乐 API 结构”，不改为完全不同的数据入口。
- 首页里涉及图片、公告文字、云音乐歌曲 ID 等壳子配置，需要能在本地后台里用结构化界面编辑，而不是只能手改代码。
- 全站卡片、背景、按钮、导航、标签、状态条统一成参考站的玻璃质感与光效语言。
- 文稿页、资料页、笔记页、专题页保留现有内容结构，只换视觉外壳与全局导航。
- 导航栏目中使用“文稿”，不使用“视频稿”。

## 非目标

- 本轮不把资料页、文稿页、笔记页的正文结构重排成参考站文章页样式。
- 本轮不把项目迁移成 Next.js 单体。
- 本轮不替换现有 GitHub Pages 静态发布思路。
- 本轮不改动资料内容录入、双语整理和引用体系本身。

## 信息架构

### 全站主导航

桌面端与移动端共享同一套站点栏目：

- 首页
- 文稿
- 资料库
- 笔记
- 专题
- 关于

如需要保持和参考站更接近的横排密度，可在实现时增加一个“归档”入口，但默认先使用上面这 6 项。

### 适配原则

- 首页按参考站结构重建。
- 其他公开页面复用同一套顶部导航、背景层、卡片材质与按钮系统。
- 页面路径不改，优先保证现有内容链接稳定。

## 视觉方向

### 整体气质

参考站的关键视觉特征不是“普通玻璃卡片”，而是以下元素组合：

- 全屏背景图或背景轮播
- 半透明滤镜叠层
- 蓝紫至粉紫的流动光晕
- 高圆角玻璃卡片
- 明显的 blur、轻描边、体积阴影
- 悬浮/显隐/旋转类动效

当前站点将从暖色纸面风格切换为偏蓝紫玻璃科幻氛围，但正文可读性保持克制，不在阅读区堆过量动态特效。

### 色彩系统

新主题变量以如下方向为准：

- 主强调色：靛蓝、蓝紫
- 次强调色：淡紫、粉紫
- 中性色：雾白、灰蓝、深夜蓝
- 阴影：柔和冷色阴影
- 边框：半透明浅色边框

### 材质系统

全站公共卡片使用统一材质：

- `border-radius` 大圆角
- 半透明背景
- `backdrop-filter: blur(...)`
- 弱描边
- 柔和阴影
- hover 轻微抬升或缩放

此系统将覆盖：

- 首页卡片
- 文稿卡片
- 资料卡片
- 笔记卡片
- 搜索结果面板
- 侧栏信息块
- 状态条
- 引用/附件/标签容器

## 首页结构设计

首页按参考站真实结构重建，阅读顺序如下。

### 1. 开场欢迎层

首页初次进入时显示简短欢迎/显现层，再切入主页主体。

要求：

- 仅在首页触发。
- 不在每次页面跳转重复完整打断式欢迎。
- 文案换成本站定位，突出“文稿、资料归档、个人素材库”。

### 2. 顶部导航区

- 桌面端：固定悬浮式导航条。
- 左侧：站名锁定块，采用“名称 + 分隔符 + 副标题”的参考站气质。
- 右侧：横排栏目。
- 滚动行为：向下滚动时隐藏，向上时重新显示。
- 当前栏目：高亮文字 + 小型激活指示。

### 3. 搜索条

导航下方保留参考站的大搜索条布局与材质。

搜索对象：

- 文稿
- 资料库公开页
- 公开笔记

搜索体验：

- 即时过滤
- 毛玻璃下拉结果面板
- 高亮命中关键词
- 点击直接进入目标页

### 4. 第一行双卡

#### 左侧大型个人/站点卡

复刻参考站的大信息卡布局，但内容改为本站语境：

- 站点名称
- 一段站点定位说明
- 公开文稿数
- 公开资料数
- 公开笔记数
- 可选的社交/仓库入口

此卡承担“欢迎页主身份卡”的作用。

#### 右侧首页音乐大卡

保留参考站核心体验：

- 封面图
- 歌曲标题与歌手信息
- 播放/暂停/切歌
- 进度条
- hover 发光感
- 播放时封面旋转

### 5. 独立歌词条

第一行双卡下方保留参考站独立长条歌词区。

行为：

- 有歌词时显示当前滚动歌词。
- 无歌词时显示当前播放状态或站点欢迎文案。
- 保留呼吸感和轻量动态反馈。

### 6. 第二行拼贴区

版式遵循参考站“左侧一块 + 右侧上下组合”的拼贴逻辑，但内容映射为本站内容：

#### 左侧主卡

- 最新文稿
- 或当前主文稿

#### 右上大横幅卡

- 当前专题入口
- 或资料库入口主推荐

#### 右下双卡

- 最新笔记
- 主题切换块或归档状态块

此处的目标是保持参考站首页节奏与版式识别，而非复用其业务内容。

### 7. 底部状态条

保留参考站横向状态面板的视觉逻辑，替换内容为本站指标：

- 当前时间
- 建站运行时间
- 文稿数量
- 资料数量
- 最近更新日期
- GitHub Pages / Astro 等运行标识

## 导航与移动端菜单

### 桌面端导航

- 固定在顶部
- 半透明毛玻璃底
- 支持滚动显隐
- 当前页面有激活态

### 移动端导航

尽量复刻参考站高辨识度菜单逻辑，而不是普通抽屉：

- 右侧悬浮触发条/按钮
- 点击后出现全屏蒙层
- 中央出现圆盘式菜单
- 菜单项分布在圆盘周围
- 可关闭

适配要求：

- 使用本站栏目名
- 处理较长中文标签，防止按钮拥挤
- 当前页高亮明显

## 音乐系统设计

### 核心约束

保留参考站的“云音乐 API 结构”。

这意味着：

- 仍然使用类似 `cloudMusicIds` 的配置入口。
- 仍然通过站内 API 或兼容层获取曲目信息。
- 保留歌词、封面、歌曲信息、播放状态这套结构化数据流。

本轮不改成完全脱离该结构的本地静态音频清单方案。

### 本站实现方式

由于当前站点是 Astro 静态站，需要引入一个局部客户端音乐层。建议实现为：

- 站点配置中保存 `cloudMusicIds`
- 新增或兼容一个客户端可访问的音乐数据接口
- 前端播放器消费统一结构的数据：
  - id
  - title
  - artist
  - cover
  - src
  - lrc / parsedLyrics

### 行为模式

- 首页显示大播放器卡 + 歌词条
- 非首页显示右下角悬浮小播放器
- 首页不重复显示悬浮小播放器
- 保留播放/暂停/切歌/进度拖动

### 风险与处理

外部音乐接口可能不稳定，因此需要：

- 加载中状态
- 加载失败状态
- 无歌词降级
- 无曲目降级

首页在接口异常时仍应保持布局完整，不因音乐失败而塌掉。

## 本地后台配置要求

### 目标

首页壳子中这些高频可变内容不能只写死在代码里，必须能在本地后台里改：

- 背景图片 / 首页横幅图等图片地址
- 公告文字 / 欢迎语 / 首页提示文字
- 云音乐歌曲 ID 列表
- 云音乐备用封面等基础音乐展示配置

### 交互形式

不满足于纯 JSON 文本编辑，需要提供结构化操作界面，风格参考用户指定博客的本地后台体验：

- 文本输入框用于公告、标题、副标题
- 可增删排序的列表用于背景图片与歌曲 ID
- 图片地址字段可配合资源页一起使用
- 保存后直接落到项目配置文件中，保持 Git 管理

### 实现约束

- 继续使用现有本地后台，不另起新管理系统
- 优先复用当前 `manager` 工程的内容与页面配置接口
- 静态站前台在构建时读取该配置文件
- 保持 GitHub Pages 静态发布兼容，不依赖线上动态数据库

## 全站适配策略

### 首页

优先最大程度贴近参考站。

### 文稿页 / 资料页 / 笔记页 / 专题页

保留现有信息架构，但统一以下外观：

- 顶部导航
- 页面背景
- 卡片材质
- 按钮样式
- 标签样式
- 搜索与状态条风格

### 正文区

正文保持高可读性：

- 不引入过强背景干扰
- 不在正文中加入高频动态粒子
- 引用块与附件块可采用玻璃卡片化，但文字对比度优先

## 技术设计

### 基础实现原则

- 保持 Astro 为主框架
- 通过小型客户端组件承接需要浏览器状态的部分
- 尽量把视觉壳、音乐状态、导航状态拆成独立组件
- 全局主题变量集中在样式入口统一管理

### 预计新增或改造的区域

- `BaseLayout.astro`
- `src/pages/index.astro`
- 全局样式文件
- 新导航组件
- 新首页组件组
- 音乐相关客户端组件

### 推荐组件边界

- `SiteChrome`：全站外壳与背景层
- `TopNav`：桌面端顶部导航
- `MobileRadialMenu`：移动端圆盘菜单
- `HomeSearch`：首页搜索
- `HomeProfileCard`：首页站点卡
- `HomeMusicCard`：首页音乐大卡
- `HomeLyricBar`：歌词长条
- `HomeFeatureGrid`：第二行拼贴区
- `StatusStrip`：底部状态条
- `MusicStore` / `MusicClient`：客户端音乐状态层
- `FloatingPlayer`：非首页悬浮播放器

### 兼容策略

- 资料页、文稿页、笔记页不要求改路由
- 内页先接入新外壳，再逐步细调卡片样式
- 尽量避免一次性改动所有内容模板

## 实施顺序

1. 重做全站外壳、背景层与主题变量
2. 重做顶部导航与移动端菜单
3. 重做首页整体骨架
4. 接入首页搜索
5. 接入云音乐 API 兼容的数据层与首页大播放器
6. 接入歌词条与悬浮播放器
7. 将内页卡片、标签、侧栏、引用块统一成新材质
8. 做桌面端与移动端逐页验证

## 验证标准

以下结果全部满足时，本轮 UI 改版视为达标：

- 首页第一眼观感明显接近参考站
- 桌面端导航和移动端菜单具有相同识别度
- 首页音乐卡与歌词条可用
- 非首页存在悬浮播放器
- 文稿页、资料页、笔记页结构未被破坏
- 全站卡片、背景、按钮、标签气质统一
- GitHub Pages 静态输出路径不被破坏
- 本地后台可以编辑首页图片、公告文字与云音乐歌曲 ID

## 已确认决策

- 采用视觉等效重建，不直接整段移植参考站前台代码
- 首页完整照参考站的结构与气质重建
- 导航栏目使用“文稿”
- 内容页保留原结构，只统一外壳
- 保留云音乐 API 结构
- 首页壳子配置需要通过本地后台结构化编辑
