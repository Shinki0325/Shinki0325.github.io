# 个人归档型博客重建设计

## 1. 背景

当前仓库 `Shinki0325.github.io` 保存的是已经生成好的静态页面，而不是可维护的博客源码。现状存在以下问题：

- 页面内容已经固化为静态 HTML，后续很难继续写作和维护。
- 评论等外部依赖已经失效，旧站不再适合作为长期可维护方案。
- 站点缺少对“公开内容”和“私有资料/草稿”的清晰边界。
- 用户希望未来主要通过网页端进行写作和整理，而不是本地命令行发布。

本次重建的目标不是简单修复旧站，而是把它升级成一个长期可维护的“个人归档型站点”：

- 既能发布正式长文。
- 也能沉淀短笔记、摘录、灵感和图片素材。
- 以 GitHub 为主要基础设施。
- 尽量减少第三方依赖。

## 2. 目标

### 2.1 产品目标

- 站点既是公开博客，也是长期可用的个人想法与素材归档入口。
- 同时支持长文和短笔记两种主要内容形态。
- 支持图文混排，满足基础排版需求。
- 支持草稿和仅自己可见的私有资料，不强迫所有内容公开。
- 用户未来主要通过浏览器完成内容新增、编辑、上传图片和发布。

### 2.2 工程目标

- 新站必须基于真正可维护的源码项目，而不是生成后的 HTML。
- 公开站点部署尽量只依赖 GitHub Pages。
- 公开内容构建稳定、依赖少、便于以后继续改版。
- 私有内容与公开内容必须在仓库结构和发布流程上明确隔离。

## 3. 约束与前提

- 用户接受更换旧博客方案，不要求保留旧页面风格。
- 用户只要求保留文章内容和基础页面能力。
- 用户希望主要用网页端发文。
- 用户当前倾向尽量只依赖 GitHub Pages，而不额外引入独立 CMS 后端。
- 用户接受“通过 GitHub 登录/编辑仓库内容，再自动发布”的工作方式。
- 一部分内容需要只自己可见，不能混入公开 Pages 产物。

## 4. 方案对比

### 4.1 方案 A：Astro + 私有源码仓库 + 公共 Pages 仓库

优点：

- 站点最终是纯静态页面，适合 GitHub Pages。
- 前端灵活，适合做“长文 + 短笔记 + 专题归档”混合结构。
- 内容可以用 Markdown 管理，长期维护成本低。
- 可以通过私有源码仓库天然隔离草稿和私有资料。
- 用户可以直接在 GitHub 网页或 `github.dev` 中写作和上传图片。

缺点：

- 不是传统意义上的独立 CMS 后台。
- 需要通过仓库结构和发布规则控制公开范围。

### 4.2 方案 B：Astro + Decap CMS + GitHub

优点：

- 写作体验更接近可视化后台。
- 适合非技术写作流程。

缺点：

- GitHub 后端认证需要额外服务端支持。
- 不再符合“尽量只依赖 GitHub Pages”的当前约束。

### 4.3 方案 C：Hugo + 双仓库结构

优点：

- 轻量、成熟、静态构建快。

缺点：

- 页面层扩展体验弱于 Astro。
- 后续如果强化卡片流、专题页、内容组件，开发体验不如 Astro。

## 5. 选型结论

采用 **方案 A：Astro + 私有源码仓库 + 公共 GitHub Pages 发布仓库**。

推荐原因：

- 最符合当前“稳定、少依赖、可长期维护”的目标。
- 能兼顾公开内容发布与私有资料沉淀。
- 未来如果确实需要更完整的可视化后台，可以在这个结构之上逐步增加，而不是推倒重来。

## 6. 总体架构

### 6.1 仓库结构

重建后采用双仓库模式：

- `private source repo`
  - 私有仓库。
  - 保存 Astro 源码、内容源文件、图片、草稿、私有资料。
  - 用户日常主要在这个仓库里写作和整理。
- `public pages repo`
  - 公开仓库。
  - 用于 GitHub Pages 对外发布。
  - 仅保存构建后的静态站点产物。
  - 现有 `Shinki0325.github.io` 仓库继续承担这一角色。

### 6.2 发布关系

- 用户在私有源码仓库中编辑内容。
- GitHub Actions 在私有仓库中执行构建。
- 构建流程只提取允许公开的内容。
- 构建后的静态文件自动推送到 `Shinki0325.github.io`。

### 6.3 公开/私有边界

公开内容：

- 长文
- 短笔记
- 专题页
- About / Links 等基础页面

私有内容：

- 草稿
- 永久私藏资料
- 未整理素材
- 不应公开的图片与摘录

公开/私有的隔离必须在目录结构和发布脚本两层同时生效，避免误发布。

## 7. 信息架构

新站不采用传统单列表博客结构，而采用“公开展示 + 归档入口”的结构。

### 7.1 公开站点页面

- `首页`
  - 最新长文
  - 最近笔记
  - 专题入口
- `长文`
  - 展示正式文章、随笔、整理型内容
- `短笔记`
  - 展示卡片式记录、摘录、灵感、图文碎片
- `专题`
  - 用户主动组织的主题内容入口
- `归档`
  - 按时间浏览全部公开内容
- `标签`
  - 按标签检索公开内容
- `关于`
  - 个人简介、站点说明
- `链接/收藏`
  - 公开分享的资源入口

### 7.2 私有内容区

私有内容不进入公开站点路由，而是只存在私有源码仓库内，作为个人资料库使用。

建议目录：

- `drafts/`
- `vault/`
- `assets/private/`

## 8. 内容模型

### 8.1 长文 Article

适合正式表达一个完整主题。

建议字段：

- `title`
- `date`
- `summary`
- `tags`
- `topics`
- `cover`
- `draft`
- `content`

表现特征：

- 正式详情页
- 支持目录
- 支持封面图
- 支持摘要和相关推荐

### 8.2 短笔记 Note

适合碎片想法、摘录、观察、临时记录、文字+图片素材沉淀。

建议字段：

- `title`（可选）
- `date`
- `tags`
- `images`
- `source`（可选）
- `draft`
- `content`

表现特征：

- 卡片式展示
- 支持无标题发布
- 支持多图
- 适合轻量归档

### 8.3 专题 Topic

适合将分散内容主动整理为主题入口页。

建议字段：

- `title`
- `description`
- `cover`
- `pinned`
- `content`

用途：

- 不是简单分类，而是人工整理后的策展页
- 用于组织某一研究方向、兴趣主题、资料合集

### 8.4 标签与专题的关系

- `tag`：细粒度检索维度
- `topic`：主动组织过的主题维度

两者并存，避免站点只能按时间阅读。

## 9. 图片与素材管理

站点需要支持“文字 + 图片资料”的长期沉淀，因此图片不能继续大量依赖外链。

### 9.1 原则

- 公开内容图片尽量本地化到仓库中。
- 私有图片不得进入公开构建流程。
- 图片目录应与内容结构对应，便于后续迁移和维护。

### 9.2 建议划分

- `public assets`
  - 给长文、短笔记、专题使用
- `private assets`
  - 给草稿、私藏资料使用

### 9.3 排版能力

MVP 需要支持：

- Markdown 正文
- 标题层级
- 引用块
- 分割线
- 列表
- 表格
- 图片说明
- 图库/多图
- 长文目录

可选增强：

- 摘录块
- 资料卡组件
- 图片组组件

## 10. 用户使用流程

### 10.1 日常写作流程

1. 打开私有源码仓库。
2. 在 GitHub 网页或 `github.dev` 中新建/编辑内容文件。
3. 根据内容性质选择：
   - `Article`
   - `Note`
   - `Topic`
4. 上传配图到对应公开或私有目录。
5. 若内容暂不公开：
   - 放入 `drafts/`
   - 或设为 `draft: true`
6. 提交变更。
7. 自动构建并发布公开内容。

### 10.2 认知负担控制

用户只需要记住两种主要公开内容：

- 正式整理过的内容写成 `长文`
- 碎片化想法和素材写成 `短笔记`

这样既保留表达自由，也不会让写作入口过多。

## 11. 迁移范围

### 11.1 需要迁移

- 旧站文章正文
- `about`
- `link`
- 适合作为基础页的静态内容

### 11.2 不迁移

- 旧评论系统
- 已失效的 Waline 服务依赖
- 已失效或不稳定的外部展示模块
- 直接复用旧生成 HTML 作为长期源码

### 11.3 迁移原则

- 以内容为核心迁移，而不是保留旧页面结构。
- 迁移时优先保正文、元信息、图片。
- 页面表现全部在新源码体系中重建。

## 12. MVP 范围

第一版只实现以下能力：

- Astro 新站源码
- 长文内容模型
- 短笔记内容模型
- 专题页
- 标签页
- 归档页
- About 基础页
- 本地图片管理
- 私有草稿与私有资料隔离
- GitHub Actions 自动发布到 Pages

## 13. 暂不纳入 MVP

- 真正独立的在线 CMS 后台
- 评论系统
- 复杂全文搜索
- 私有内容在线登录查看
- 云端图片管理
- 多用户协作写作

原因：

- 当前首要目标是长期可用、好维护、边界清晰。
- 过早增加外部依赖会重新引入未来失效风险。

## 14. 风险与对应策略

### 14.1 网页端写作体验不如独立 CMS

策略：

- 先以 GitHub 网页编辑和 `github.dev` 作为 MVP 写作入口。
- 后续如确实成为明显痛点，再补 CMS 层。

### 14.2 私有内容误公开

策略：

- 在目录层隔离。
- 在构建层白名单发布。
- 在 CI 中增加校验，阻止私有目录进入公开产物。

### 14.3 旧内容迁移格式不统一

策略：

- 按“文章优先、页面其次”的顺序迁移。
- 允许部分旧内容先简化导入，再逐步整理元信息。

## 15. 后续实施阶段建议

建议按以下阶段推进：

1. 初始化私有源码仓库和 Astro 项目结构。
2. 建立 `Article` / `Note` / `Topic` 内容集合。
3. 完成首页、长文页、笔记页、专题页、归档页。
4. 配置 GitHub Actions 发布到 `Shinki0325.github.io`。
5. 迁移旧文章与基础页面。
6. 补充图片整理与样式优化。

## 16. 验收标准

完成后应满足：

- 用户可以在浏览器中新增长文和短笔记。
- 用户可以上传图片并在内容中稳定引用。
- 草稿和私有资料不会进入公开站点。
- 公开站点可以稳定部署到 GitHub Pages。
- 旧博客的主要文章内容已迁移到新体系中。
- 首页能够同时体现正式文章、短笔记和主题归档能力。
