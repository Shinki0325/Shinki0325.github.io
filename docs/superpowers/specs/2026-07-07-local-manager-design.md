# 本地后台与静态前台整合设计

## 1. 背景

当前仓库 `D:/blog` 已经完成了第一轮博客重建：

- 公开站点使用 Astro 构建并发布到 GitHub Pages
- 内容已经迁移到可维护的 Markdown / Astro 结构
- 草稿与私有内容已经和公开产物分离

用户的新目标不是继续维持一个“纯手工改文件”的静态博客，而是在保留静态发布优势的前提下，尽量获得接近参考项目 `heiehiehi/XinghuisamaBlogs` 的内容管理体验：

- 前台继续静态发布
- 本地运行网页后台
- 文章和笔记使用 Markdown 写作
- 首页等页面支持多内容区块编辑
- 草稿与私有内容继续隔离

这意味着需要在当前 Astro 站点之上新增一层本地内容管理系统，而不是整体迁移到新的线上架构。

## 2. 目标

### 2.1 产品目标

- 保留当前博客作为公开展示站点
- 增加一个仅本机运行的网页后台，负责写作和内容整理
- 让文章、笔记、草稿的编辑体验明显优于直接改文件
- 让首页、链接页、专题入口等页面支持固定模板块编辑
- 让图片插入、草稿保存、发布前检查和推送动作尽量集中在后台完成

### 2.2 工程目标

- 继续保持公开站点为纯静态产物
- 继续兼容 GitHub Pages 发布链路
- 不引入线上数据库和远程 CMS
- 内容源仍然保持可读、可迁移、可手工维护
- 后台只是内容控制台，不成为唯一数据来源

## 3. 约束与前提

- 用户最终接受的路线是：保留 Astro 静态前台，新增本地网页后台
- 本地后台是单用户、本机使用，不做登录系统
- 文章正文与笔记正文保留 Markdown 作为主格式
- 页面搭建使用 JSON 配置和固定模板块，不做完全自由布局
- 固定模板块采用“增删、排序、填字段”的模式，不做任意组件开发器
- 草稿和私有资料必须继续排除在公开构建产物之外
- 当前工作目录已经是可信仓库，可继续在此仓库内演进

## 4. 方案结论

采用以下整体方案：

- 前台：继续使用 Astro 作为公开站点渲染层
- 后台：在同一仓库内新增一个本地运行的网页管理应用 `manager`
- 内容源：正文使用 Markdown，页面配置使用 JSON，图片使用本地资源目录
- 发布方式：后台触发本地校验、构建、Git 提交与推送，GitHub Actions 继续负责 Pages 部署

这是当前最平衡的路线，因为它同时满足：

- 不打断已上线站点
- 尽量保留参考项目中“好用的本地后台”这一价值
- 不把内容锁死在数据库或远程 CMS 中
- 后续可以渐进增强，而不是整体推翻重来

## 5. 总体架构

系统分为三层。

### 5.1 `site`

当前仓库中的 Astro 前台，负责：

- 首页
- 文章页
- 笔记页
- 专题页
- 归档页
- 关于页
- 链接页

`site` 的唯一输入是仓库中的内容文件和页面配置文件，唯一输出是静态站点产物。

### 5.2 `content source`

内容源保留为普通文件：

- `Markdown`：文章、笔记、草稿
- `JSON`：首页区块、导航、链接数据、专题入口、全站展示配置
- `assets`：图片、封面、附件

这一层是系统的真实数据源。无论前台还是后台，都只围绕这一层读写。

### 5.3 `local manager`

本地网页后台运行在 `localhost`，仅供用户本人使用。

它由两部分组成：

- 浏览器中的管理界面
- 仅监听本机回环地址的本地文件服务与命令服务

之所以需要本地服务，是因为纯前端页面本身不能稳定地直接读写仓库文件、复制图片、执行构建和调用 Git。第一期明确采用“本地网页界面 + 本地服务”的结构，不做远程 API，不对局域网开放。

它负责：

- 新建和编辑文章、笔记、草稿
- Markdown 编辑与实时预览
- 可视化编辑 frontmatter
- 可视化编辑首页和页面区块
- 选择、复制和引用本地图片
- 执行构建前校验
- 触发本地构建
- 触发 Git 提交和推送

后台不是在线服务，不对外网开放，也不要求部署到服务器。

## 6. 内容模型

### 6.1 Markdown 内容集合

建议继续沿用当前 Astro collection 方向，并补强字段。

第一期遵循“兼容现有 collection，渐进补强 frontmatter”的原则，不一次性推翻当前字段设计。

当前仓库同时存在 `drafts` collection 和公开 collection 中的 `draft` 布尔字段。第一期后台以 `drafts` 目录作为主要草稿入口，`articles` / `notes` 中的 `draft` 字段继续保留兼容，但不再作为后台写作流程的首选存储方式。

#### `articles`

适合正式文章、长文、系统整理内容。

建议字段：

- `title`
- `summary`
- `date`
- `tags`
- `topics`
- `cover`
- `draft`
- `pinned`
- `template`

#### `notes`

适合短内容、灵感、摘录、图文记录。

建议字段：

- `title`
- `date`
- `summary`
- `tags`
- `images`
- `source`
- `draft`
- `pinned`

#### `drafts`

适合未发布文章或未整理内容，仅本地后台可见。

建议字段：

- `title`
- `date`
- `summary`
- `kind`
- `tags`
- `cover`

`drafts` 不进入公开构建，也不进入公开索引。

### 6.2 JSON 页面配置

以下页面逐步从硬编码展示数据迁移为 JSON 配置驱动：

- 首页
- 关于页
- 链接页
- 专题入口页
- 导航
- 页脚

建议建立独立配置目录，例如：

- `src/config/pages/home.json`
- `src/config/pages/about.json`
- `src/config/pages/links.json`
- `src/config/site/navigation.json`
- `src/config/site/meta.json`

当前仓库里已经存在 `src/data/site.ts`。第一期允许保留旧文件并做渐进迁移，但新的页面配置能力统一落到 `src/config` 目录，不再继续向 `src/data` 增长同类数据。

### 6.3 固定模板块

第一期只支持固定模板块，不做自由搭积木。

建议首批支持的区块类型：

- `hero`
- `article-list`
- `note-list`
- `topic-grid`
- `link-grid`
- `image-gallery`
- `rich-text`
- `quote`

每个区块都有清晰 schema。后台只允许：

- 添加区块
- 删除区块
- 调整顺序
- 编辑字段

不允许随意写任意 HTML 或注入不受控布局。

### 6.4 文章内增强块

正文仍然以 Markdown 为主。文章内增强块不是第一期必做项。

第一期只保证：

- 普通 Markdown 写作
- 本地图片插入
- 基础封面和摘要字段编辑

如果后续确认确实需要，再在第二期评估是否补充受控增强块，例如图片组、引用卡或提示块，但它们不应替代 Markdown 主体。

## 7. 目录与数据组织

建议在当前仓库中形成更明确的内容分层。

### 7.1 内容目录

- `src/content/articles`
- `src/content/notes`
- `src/content/drafts`
- `src/content/topics`
- `src/content/vault`

### 7.2 配置目录

- `src/config/pages`
- `src/config/site`

### 7.3 资源目录

- `public/uploads/articles`
- `public/uploads/notes`
- `public/uploads/pages`

如果后续需要更强的本地素材沉淀，可以在仓库内增加非公开素材目录，但第一期只处理公开资源与草稿引用场景。

## 8. 本地后台功能设计

第一期后台建议包含 5 个主页面。

### 8.1 Dashboard

展示：

- 最近文章
- 最近草稿
- 最近笔记
- 最近构建结果
- 最近 Git 状态

提供快捷操作：

- 新建文章
- 新建笔记
- 新建草稿
- 构建站点
- 提交推送

### 8.2 Content

内容列表页，支持切换：

- 文章
- 笔记
- 草稿

能力：

- 搜索
- 标签筛选
- 状态筛选
- 新建
- 删除
- 编辑
- 置顶

### 8.3 Editor

用于 Markdown 编辑。

必须具备：

- Markdown 文本编辑
- 实时预览
- frontmatter 表单化编辑
- 封面图选择
- 标签编辑
- 草稿 / 公开状态切换
- 自动保存
- 插入图片

### 8.4 Page Builder

用于编辑 JSON 驱动页面。第一期必须完成首页编辑，其他页面先保留配置结构与后续扩展位。

能力：

- 选择页面
- 查看区块列表
- 新增区块
- 删除区块
- 调整顺序
- 编辑字段
- 页面预览

### 8.5 Assets

用于管理图片资源：

- 浏览现有上传内容
- 复制新图片到统一目录
- 返回可插入正文或配置文件的相对路径

站点标题、描述、导航等设置在第一期只需要支持最小可编辑范围；关于页和链接页的可视化编辑不进入第一期核心范围。

## 9. 写作与发布流程

### 9.1 文章写作流程

1. 在后台创建文章
2. 填写标题、摘要、标签、封面
3. 编写 Markdown 正文
4. 插入图片
5. 保存为草稿或发布

### 9.2 页面编辑流程

1. 进入 Page Builder
2. 第一期开启首页配置页
3. 添加 / 删除 / 调整固定模板块
4. 编辑区块字段
5. 保存 JSON 配置

### 9.3 发布流程

1. 后台执行内容校验
2. 后台执行本地构建
3. 构建成功后执行 Git 提交
4. 推送到远端
5. GitHub Actions 完成 Pages 部署

### 9.4 草稿与私有内容隔离

后台必须清楚区分：

- `drafts`：未发布内容
- `vault`：长期私有内容
- `public content`：允许构建和部署的内容

构建前校验必须继续阻止草稿和私有条目进入公开产物。

## 10. 技术路线

### 10.1 前台

继续使用当前 Astro 站点，不整体迁移框架。

### 10.2 后台

新增仓库内的本地网页应用，采用单独子应用实现，例如：

- `manager/`

`manager/` 内同时包含：

- 本地浏览器管理界面
- 本地文件与命令服务

后台负责：

- 本地文件读写
- 内容 schema 校验
- 调用构建命令
- 调用 Git 命令

### 10.3 共享层

建议抽出共享模块供前后台共用：

- 内容 schema
- 路径工具
- 页面区块 schema
- 站点配置 schema
- 发布前校验逻辑

这样可以避免前后台对内容格式理解不一致。

## 11. 第一期开工范围

第一期必须完成：

- 本地后台骨架
- 文章 / 笔记 / 草稿 列表与编辑
- Markdown 编辑器与预览
- frontmatter 表单编辑
- 首页区块 JSON 编辑
- 固定模板块增删排序
- 本地图片插入
- 构建校验
- 本地构建
- Git 提交与推送入口

第一期暂不做：

- 多用户系统
- 远程后台
- 数据库
- 完全自由布局
- 文章内可视化区块编辑器
- 关于页 / 链接页 / 全站设置的完整可视化后台
- 强制图床集成
- 复杂权限和登录

## 12. 风险与应对

### 12.1 后台过重，反而难维护

应对：

- 第一版只做单用户、本地使用
- 只做固定模板块
- 不引入数据库

### 12.2 前后台 schema 不一致

应对：

- 抽出共享 schema 层
- 所有页面 JSON 和 Markdown frontmatter 都经过同一套校验

### 12.3 图片管理混乱

应对：

- 统一上传目录
- 后台自动复制并生成相对路径
- 按内容类型划分资源目录

### 12.4 草稿泄漏到公开站

应对：

- 保留现有公开内容过滤
- 保留构建后断言脚本
- 后台发布动作必须调用同一套检查

## 13. 实施阶段建议

建议按以下阶段推进：

1. 补齐共享 schema 与配置目录结构
2. 新增本地后台项目骨架
3. 接入文章 / 笔记 / 草稿 的列表与编辑
4. 接入首页区块编辑
5. 接入图片管理
6. 接入构建 / 提交 / 推送动作
7. 最后统一重做前台风格，使其更接近参考博客气质

最后一步放在后面，是为了先把内容管理打通，再做视觉升级，避免“外观很像但不能长期用”。

## 14. 验收标准

完成后应满足：

- 用户可以在本地网页后台新建文章、笔记、草稿
- 用户可以在后台编辑 Markdown 并看到预览
- 用户可以在后台管理首页区块
- 用户可以在后台插入本地图片
- 用户可以通过后台执行构建和推送
- 草稿和私有内容不会进入公开站点
- 公开站点仍然稳定部署到 GitHub Pages
- 后台失效时，内容文件依然可以手工维护
