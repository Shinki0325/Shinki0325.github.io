# 视频稿归档与资料库整合设计

## 1. 背景

当前仓库 `D:/blog` 已经完成了第一轮博客重建：

- 公开站点使用 Astro 构建并发布到 GitHub Pages
- 内容已经迁移到 Markdown / Astro 结构
- 草稿与私有内容已经和公开产物分离

用户现在的最终目标，不只是“恢复一个能发文章的博客”，而是把这个站点变成一个长期可维护的个人归档系统：

- 公开展示视频稿、文章和整理内容
- 归档大量杂来源参考资料
- 让主稿可以像 wiki 一样链接到资料页
- 让资料页本身也能形成互相引用和回链
- 同时保留私有资料与草稿，不进入公开站点

因此，这个项目的本质不再是单纯的博客 CMS，而是“静态博客前台 + 本地写作后台 + 公开资料库 + 私有知识库”的混合体系。

## 2. 目标

### 2.1 产品目标

- 让站点成为视频稿、文章和参考资料的长期归档库
- 让公开主稿可以链接到公开资料页，形成类似 wiki 的阅读体验
- 让参考资料既能作为单条来源页存在，也能整理成专题页
- 让用户在本地网页后台里完成主稿写作、资料整理、图片插入和发布
- 让私有资料和草稿继续保留，只在本地后台和本地预览可见

### 2.2 工程目标

- 继续保持公开站点为纯静态产物
- 继续兼容 GitHub Pages 发布链路
- 不引入线上数据库和远程 CMS
- 内容源继续保持为可读、可迁移、可手工维护的文件
- 后台只是写作与整理工具，不成为唯一数据来源

## 3. 约束与前提

- 最终路线仍然是：保留 Astro 静态前台，新增本地网页后台
- 本地后台是单用户、本机使用，不做登录系统
- 正文继续以 Markdown 为主格式
- 页面搭建继续使用 JSON 配置和固定模板块，不做完全自由布局
- 参考资料统一作为一个内容集合管理，用 `kind` 区分不同资料形态
- 视频主稿继续保留在 `articles` 集合内，不额外拆出 `scripts` 集合
- 公开主稿不能链接私有资料
- 私有资料、草稿和未公开附件必须继续排除在公开构建产物之外

## 4. 方案结论

采用以下整体方案：

- 前台：继续使用 Astro 作为公开站点渲染层
- 后台：在同一仓库内新增一个本地运行的网页管理应用 `manager`
- 内容源：主稿、资料页和草稿使用 Markdown，页面配置使用 JSON，图片与截图使用本地资源目录
- 发布方式：后台触发本地校验、构建、Git 提交与推送，GitHub Actions 继续负责 Pages 部署

这条路线的定位不是“把博客换个后台”，而是把公开站点底层内容模型升级成：

- `主稿`
- `公开资料页`
- `私有资料页`
- `内部链接`
- `自动回链`

这样既能保留现有静态发布优势，又能获得接近个人 wiki 的归档能力。

## 5. 总体架构

系统分为三层。

### 5.1 `site`

当前仓库中的 Astro 前台，负责：

- 首页
- 文章页
- 视频稿页
- 资料库索引页
- 资料详情页
- 专题页
- 标签 / 主题聚合页
- 关于页与基础静态页面

`site` 的唯一输入是仓库中的内容文件和页面配置文件，唯一输出是静态站点产物。

### 5.2 `content source`

内容源保留为普通文件：

- `Markdown`：文章、视频稿、资料页、草稿、私有资料
- `JSON`：首页区块、导航、站点元信息、展示配置
- `assets`：图片、截图、封面、附件

这一层是真实数据源。无论前台还是后台，都只围绕这一层读写。

### 5.3 `local manager`

本地网页后台运行在 `localhost`，仅供用户本人使用。

它由两部分组成：

- 浏览器中的管理界面
- 仅监听本机回环地址的本地文件服务与命令服务

之所以需要本地服务，是因为纯前端页面本身不能稳定地直接读写仓库文件、复制图片、执行构建和调用 Git。第一期明确采用“本地网页界面 + 本地服务”的结构，不做远程 API，不对局域网开放。

它负责：

- 新建和编辑主稿、资料页、草稿
- Markdown 编辑与实时预览
- 可视化编辑 frontmatter
- 搜索并插入站内资料链接
- 管理图片和截图附件
- 查看自动回链结果
- 执行构建前校验
- 触发本地构建
- 触发 Git 提交和推送

后台不是在线服务，不对外网开放，也不要求部署到服务器。

## 6. 内容模型

### 6.1 Markdown 内容集合

第一期遵循“兼容现有 collection，渐进补强 frontmatter”的原则，不一次性推翻当前结构。

#### `articles`

`articles` 继续承载公开长文与视频主稿，通过字段区分类型。

建议字段：

- `title`
- `summary`
- `date`
- `type`
- `tags`
- `topics`
- `cover`
- `draft`
- `pinned`
- `template`

其中：

- `type: script` 表示视频稿
- `type: article` 表示普通文章

视频稿正文允许引用公开资料页，但不允许引用私有资料页。

#### `references`

新增统一的资料集合，作为公开资料库和本地资料库的核心。

建议基础字段：

- `title`
- `kind`
- `visibility`
- `date`
- `summary`
- `tags`
- `topics`
- `cover`
- `attachments`
- `aliases`
- `draft`

其中：

- `kind: source | topic`
- `visibility: public | private`

`references` 中的两类条目定义如下：

- `source`：单条来源页，例如网页、书摘、截图出处、视频片段出处、聊天记录摘录
- `topic`：专题整理页，用来归并多条资料并写出你的整理结论

`references` 中允许存在 `visibility: private` 的条目，它们的定位是“仍然属于资料库体系、未来可能转公开、也需要参与链接和整理”的私有资料。

#### `notes`

`notes` 继续保留，适合短想法、碎片灵感和轻量图文记录。

如果某条内容最终演化成正式来源页或专题页，可以后续迁移到 `references`。

#### `drafts`

`drafts` 继续作为主要草稿入口。

主稿草稿、资料草稿和待整理条目都优先放在这里，再由后台转入公开集合或私有集合。

#### `vault`

`vault` 继续承载长期私有资料。

它只在本地后台和本地预览出现，不进入公开构建，也不能被公开主稿引用。

`vault` 和 `references.visibility: private` 的分工如下：

- `references.visibility: private`：属于资料库体系，只是暂时不公开
- `vault`：长期私藏、研究备忘或不准备公开发布的材料

### 6.2 `references` 字段细化

所有资料页默认采用“结构化卡片 + Markdown 正文”的形态。

#### `kind: source`

单条来源页除基础字段外，建议补充：

- `sourceType`
- `sourceTitle`
- `sourceUrl`
- `author`
- `publishedAt`
- `publisher`
- `quote`
- `note`

字段含义：

- `sourceType`：网页、书籍、视频、播客、图片、截图、聊天记录等
- `sourceTitle`：原始来源标题
- `sourceUrl`：原始链接
- `quote`：核心摘录
- `note`：你的总结、评价或可用点

默认保存策略为：

- 必须保留来源信息
- 通常保留摘录和你的总结
- 对特别重要的资料，允许继续写更完整的整理正文

#### `kind: topic`

专题整理页除基础字段外，建议补充：

- `intro`
- `relatedRefs`
- `relatedScripts`

其中：

- `intro`：专题说明
- `relatedRefs`：主动整理进来的资料页列表
- `relatedScripts`：主动关联的主稿列表，可选

专题页正文继续使用 Markdown 写你的整理和归纳。

### 6.3 JSON 页面配置

JSON 配置继续用于：

- 首页区块
- 导航
- 站点元信息
- 资料库入口页展示配置

建议目录：

- `src/config/pages/home.json`
- `src/config/pages/references.json`
- `src/config/site/navigation.json`
- `src/config/site/meta.json`

当前仓库里已经存在 `src/data/site.ts`。第一期允许保留旧文件并做渐进迁移，但新的页面配置能力统一落到 `src/config` 目录，不再继续向 `src/data` 增长同类数据。

### 6.4 固定模板块

第一期只支持固定模板块，不做自由搭积木。

建议首批支持的区块类型：

- `hero`
- `article-list`
- `reference-list`
- `topic-grid`
- `image-gallery`
- `rich-text`
- `quote`

每个区块都有清晰 schema。后台只允许：

- 添加区块
- 删除区块
- 调整顺序
- 编辑字段

不允许随意写任意 HTML 或注入不受控布局。

## 7. 链接与关系模型

### 7.1 内部链接规则

主稿和资料页都支持两种站内链接写法：

- Wiki 风格链接：`[[标题]]`
- 带显示文字的 wiki 风格链接：`[[标题|显示文字]]`
- 标准 Markdown 链接：`[显示文字](/references/slug)`

后台编辑器需要提供“搜索并插入资料链接”的能力，避免手动拼路径。

如果出现同名资料冲突，后台插入器必须要求用户明确选择目标条目。

### 7.2 可见性规则

公开主稿只能链接公开资料。

这条规则同时适用于：

- `articles` 中公开发布的内容
- `references` 中公开的专题页

以下情况必须在校验阶段报错：

- 公开主稿链接到 `visibility: private` 的资料页
- 公开资料页链接到未公开且不允许外露的目标
- 链接目标不存在

### 7.3 自动回链

第一期回链采用“自动生成，不手工维护”的方式。

实现逻辑：

- 扫描 `articles` 和 `references` 正文中的内部链接
- 为每个资料页建立反向索引
- 在资料详情页底部展示“哪些主稿 / 哪些资料页引用了我”

用户平时只需要正常写链接，不需要手工维护关系字段。

### 8.1 内容目录

- `src/content/references`
- `src/content/notes`
- `src/content/drafts`

### 8.2 配置目录

- `src/config/pages`

- 图片

PDF 或其他复杂附件类型不进入第一期核心范围。

## 9. 前台页面结构

建议形成以下公开路由结构：

- `/articles/[slug]`
- `/references/[slug]`
- `/references`
- `/tags/[tag]`
- `/topics/[topic]`

其中：

- 视频稿仍然走 `/articles/[slug]`，由模板和字段区分显示方式
- `references` 详情页更像 wiki 条目页
- `/references` 是资料库入口，不只是普通时间归档页

### 9.1 视频稿页

视频稿页主体仍然像一篇可阅读长文，但文中资料链接应当顺滑跳转到对应资料页。

### 9.2 资料详情页

资料详情页建议分三段：

- 顶部：结构化资料卡片
- 中部：摘录、总结与 Markdown 正文
- 底部：自动回链列表

### 9.3 资料库索引页

资料库索引页需要支持：

- 按标签筛选
- 按主题筛选
- 按来源类型筛选
- 区分 `source` 与 `topic`

## 10. 本地后台功能设计

第一期后台建议包含 6 个主页面。

### 10.1 Dashboard

展示：

- 最近主稿
- 最近资料页
- 最近草稿
- 最近构建结果
- 最近 Git 状态

提供快捷操作：

- 新建视频稿
- 新建普通文章
- 新建来源页
- 新建专题页
- 构建站点
- 提交推送

### 10.2 Content

内容列表页，支持切换：

- 文章 / 视频稿
- 资料页
- 草稿
- 私有资料

能力：

- 搜索
- 标签筛选
- 类型筛选
- 可见性筛选
- 新建
- 删除
- 编辑

### 10.3 Editor

用于 Markdown 编辑。

必须具备：

- Markdown 文本编辑
- 实时预览
- frontmatter 表单化编辑
- 标题、摘要、标签、主题编辑
- 封面图选择
- 可见性 / 草稿状态切换
- 自动保存
- 插入图片
- 搜索并插入内部资料链接

### 10.4 Reference Editor

资料页编辑器在通用编辑器之上增加资料专用能力：

- 选择 `kind: source | topic`
- 编辑结构化来源字段
- 编辑原始链接与来源元数据
- 录入摘录和总结
- 选择图片 / 截图附件
- 查看自动回链结果

### 10.5 Page Builder

用于编辑 JSON 驱动页面。第一期必须完成首页与资料库入口页的基础配置编辑。

能力：

- 选择页面
- 查看区块列表
- 新增区块
- 删除区块
- 调整顺序
- 编辑字段
- 页面预览

### 10.6 Assets

用于管理图片和截图资源：

- 浏览现有上传内容
- 复制新素材到统一目录
- 返回可插入正文或配置文件的相对路径

## 11. 写作与发布流程

### 11.1 视频稿写作流程

1. 在后台创建 `articles` 条目并选择 `type: script`
2. 填写标题、摘要、标签、主题、封面
3. 编写 Markdown 正文
4. 搜索并插入公开资料链接
5. 保存为草稿或发布

### 11.2 来源页整理流程

1. 在后台创建 `references` 条目并选择 `kind: source`
2. 填写来源信息、原链接、作者、发布时间等字段
3. 录入摘录
4. 写入你的总结
5. 上传图片或截图附件
6. 设置为公开或私有

### 11.3 专题页整理流程

1. 在后台创建 `references` 条目并选择 `kind: topic`
2. 填写标题、摘要、主题和标签
3. 选择相关资料页
4. 编写 Markdown 整理正文
5. 保存为草稿或发布

### 11.4 发布流程

1. 后台执行内容校验
2. 校验内部链接与可见性规则
3. 后台执行本地构建
4. 构建成功后执行 Git 提交
5. 推送到远端
6. GitHub Actions 完成 Pages 部署

## 13. 风险与应对

### 13.1 模型过重，后台难维护

应对：

- 第一版只做单用户、本地使用
- 资料集合统一为一个 `references`
- 不引入数据库

### 13.2 公开内容误链私有资料

应对：

- 明确“公开主稿不能链接私有资料”
- 构建前强制校验可见性规则
- 后台插入器默认只展示当前允许公开引用的资料

### 13.3 资料命名冲突导致链接歧义

应对：

- 支持 `aliases`
- 同名时由后台插入器强制选目标
- 构建期输出冲突报告

### 13.4 图片与截图管理混乱

应对：

- 统一上传目录
- 后台复制素材并生成相对路径
- 按文章 / 资料 / 共享目录分层

## 14. 验收标准

完成后应满足：

- 用户可以在本地网页后台新建视频稿、普通文章、来源页、专题页和草稿
- 用户可以在后台编辑 Markdown 并看到预览
- 用户可以在后台录入结构化来源信息、摘录和总结
- 用户可以在主稿里搜索并插入公开资料链接
- 公开资料页可以显示自动回链
- 资料库索引页可以按基础维度筛选
- 公开主稿不会错误链接私有资料
- 用户可以在后台插入本地图片和截图
- 用户可以通过后台执行构建和推送
- 公开站点仍然稳定部署到 GitHub Pages
- 后台失效时，内容文件依然可以手工维护
