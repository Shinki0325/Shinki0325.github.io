# 参考资料页正文阅读与对照翻译设计

## 1. 背景

当前站点已经具备公开文章页、参考资料页、主题页、笔记页与站内互链能力，也已经完成第一批九十年代 galgame 资料包的导入。

现阶段参考资料页虽然已经能展示：

- 资料摘要
- 来源信息
- 归档附件
- 截图预览
- 站内回链与关联关系

但阅读体验仍然偏“资料卡”而不是“可直接引用的资料正文页”：

- 左侧主栏现在主要显示编者摘要
- 真正的站内摘录仍位于下方单独模块
- 外文资料还没有逐段中文对照
- 页面栏目名中仍有一部分英文命名

用户希望把参考资料页进一步升级成接近 wiki 条目的形态，让视频稿、笔记和专题页引用资料时，优先落到站内可阅读正文，而不是先跳到外链或附件。

## 2. 目标

### 2.1 产品目标

- 让参考资料页本身成为可直接阅读和引用的正文页
- 让外文资料支持“原文一段，下面紧跟中文译文”的对照阅读
- 让中文资料可以直接展示整理后的正文，而不是只显示摘要
- 让主稿、笔记和专题页继续优先链接站内参考资料页
- 让页面观感更接近 wiki / 资料库，而不是附件列表页

### 2.2 维护目标

- 保持当前 GitHub Pages 静态发布路线不变
- 不引入远程翻译服务或远程数据库
- 内容仍然以可手工维护的 Markdown 为核心
- 允许资料分批整理，不要求一次性翻完全部正文
- 保持现有文章、主题页、笔记页和回链能力不受影响

## 3. 范围与非目标

### 3.1 本次范围

- 改造公开参考资料详情页
- 扩展参考资料的数据结构以支持“阅读正文块”
- 将参考资料详情页中的栏目标题和属性名称统一改为中文
- 为首批重点资料补入可展示正文与对照译文
- 为外文资料提供段落级对照渲染
- 为未整理完成的资料保留站内摘录回退能力

### 3.2 非目标

- 不做自动机器翻译接入
- 不做在线后台、账号系统或云端 CMS
- 不在本轮中重构整站内容模型
- 不要求一次性覆盖全部历史参考资料
- 不改变当前站点的静态部署方式

## 4. 设计结论

采用“参考资料页内嵌阅读正文块”的方案，并保留 `txt` 摘录作为回退。

最终每个参考资料页分为两层内容：

- 编者说明：保留现有 Markdown 正文，作为资料价值说明和使用提示
- 阅读正文：新增结构化段落块，作为页面左侧主栏的核心内容

对于外文资料，阅读正文采用以下顺序渲染：

1. 原文段落
2. 中文译文
3. 可选备注

对于中文资料，直接渲染整理后的正文段落，不额外显示译文容器。

如果某条资料暂时还没有人工整理正文，则页面回退显示站内 `txt` 摘录，确保参考页仍然具备可引用的站内文本。

## 5. 页面结构

### 5.1 总体布局

保留当前 `PostLayout` 的左右栏结构：

- 左侧主栏：正文阅读区
- 右侧边栏：资料信息与关系导航

不新增新的独立页面模板，而是在现有参考资料详情页上升级信息层次。

### 5.2 左侧主栏

左侧主栏调整为以下顺序：

1. 编者说明
2. 正文阅读
3. 回退摘录（仅在没有整理正文时出现）

对应中文界面命名建议如下：

- `编者说明`
- `正文阅读`
- `站内摘录`
- `对照翻译`

其中“正文阅读”是页面的核心区域，打开资料页后应尽量在第一屏内进入正文，而不是先看到附件或扩展说明。

### 5.3 右侧边栏

右侧保留现有结构，但统一改成中文命名。建议栏目为：

- `参考资料`
- `资料详情`
- `归档文件`
- `关联内容`
- `引用此页`

字段名也统一中文化，例如：

- `类型`
- `来源类型`
- `来源标题`
- `作者`
- `发布方`
- `发布时间`
- `标签`
- `主题`
- `来源网址`

右侧仍然承担：

- 元数据展示
- 归档文件入口
- 截图预览
- 关联参考资料
- 关联主稿
- 反向引用入口

## 6. 内容模型

### 6.1 基础原则

参考资料继续使用 Markdown 文件作为主存储格式，不新增数据库或独立内容源。

每个参考资料条目保留两类内容：

- frontmatter：结构化元信息
- Markdown 正文：编者说明

在此基础上，新增一组用于正文展示的结构化字段。

### 6.2 新增字段

建议在 `references` schema 中补充以下字段：

- `readingMode`
- `sourceLanguage`
- `translationLanguage`
- `readingBlocks`

字段语义如下：

- `readingMode`
  - `curated`：使用人工整理的阅读正文块
  - `extract`：使用现有摘录回退
- `sourceLanguage`
  - 资料原文语言，例如 `zh`、`ja`、`en`
- `translationLanguage`
  - 当前阶段固定写 `zh-CN`
- `readingBlocks`
  - 一个段落数组，用于承载可展示的正文内容

### 6.3 `readingBlocks` 结构

每个段落块建议支持以下字段：

- `label`
- `original`
- `translation`
- `note`

字段含义如下：

- `label`
  - 段落小标题或段落标签，例如“平台定位”“销量背景”
- `original`
  - 原文正文
- `translation`
  - 中文译文
- `note`
  - 编者备注、语境提示或引用提示

使用规则如下：

- 中文资料通常只写 `original`
- 外文资料写 `original` 与 `translation`
- `note` 只承载说明，不混入原文或译文内容

### 6.4 内容录入原则

- 不要求一次整理完整全文，优先整理视频稿真正会引用到的关键段落
- 还未整理完的资料也可以先公开，但要保留 `txt` 摘录作为补充
- 译文以人工录入为准，不走自动机器翻译
- 原文和译文按段对应，方便后续引用、校对和继续扩写

## 7. 渲染行为

### 7.1 有阅读正文块时

当资料存在 `readingBlocks` 且内容非空时：

- 页面左侧显示 `编者说明`
- 紧接着显示 `正文阅读`
- 每个段落块按既定顺序渲染
- 如果是外文资料，显示“原文 + 中文译文”
- 如果段落带 `note`，则以弱化样式显示“编者备注”
- 不再把 `txt` 摘录当作主要正文展示

### 7.2 无阅读正文块时

当资料没有人工整理正文时：

- 页面左侧仍显示 `编者说明`
- 下方显示 `站内摘录`
- `站内摘录` 来自现有附件中的 `txt`
- 如果附件中没有可用文本，则不显示该区块

### 7.3 文本呈现要求

- 原文与译文必须在视觉上清晰区分
- 译文始终紧跟对应原文，避免用户来回寻找
- 桌面与移动端都保持单列阅读流，不采用左右双栏对照
- 保持良好的段落间距和可引用性，不使用折叠式隐藏正文

## 8. 文案与命名规范

本次改造中，参考资料详情页的页面标题、栏目标题、属性标签统一使用中文。

命名要求如下：

- 避免中英混杂的栏目名
- 保留少量必要专有名词原文，例如作品名、站点名、出处名
- 中文标签要尽量直观，便于后续长期维护

示例：

- `Reference Details` -> `资料详情`
- `Archive Files` -> `归档文件`
- `Network Links` -> `关联内容`
- `Referenced By` -> `引用此页`
- `Site Text Extract` -> `站内摘录`

## 9. 首批落地策略

第一批优先处理已经导入且最适合展示正文的资料页，例如：

- `PC-9801-コンピュータ博物館`
- `To Heart - Wikipedia`
- `YU-NO - Wikipedia`

首批目标不是做大批量自动导入，而是先建立一套稳定、可复制的资料页样板，包括：

- 有摘要说明
- 有正文阅读块
- 有逐段对照翻译
- 有站内互链
- 有归档附件和截图

后续新的资料页可以沿用这一结构，持续扩充。

## 10. 测试与验收

### 10.1 验收标准

- 打开参考资料页后，左侧主栏优先显示可阅读正文
- 外文资料能看到“原文一段，下面紧跟中文译文”
- 中文资料能直接展示整理后的正文
- 没有整理正文的资料仍能回退显示站内摘录
- 右侧边栏继续保留资料详情、归档文件、关联内容、引用此页
- 现有文章、笔记、主题页中的站内引用不受影响
- 页面栏目名和属性名统一改为中文

### 10.2 验证方式

至少新增或更新以下验证：

- 参考资料页正文阅读区渲染测试
- 外文资料译文显示测试
- 站内摘录回退测试
- 现有参考资料页附件与回链可见性测试
- `pnpm check`
- `pnpm build`
- 关键公开页面的路由与可见性测试

## 11. 风险与取舍

### 11.1 主要风险

- 人工分段和翻译需要持续投入时间
- 不同来源的 `txt` 清洗质量不一致
- 如果一开始追求全文翻译，容易拖慢整体归档推进

### 11.2 对应取舍

- 优先整理会被主稿引用的关键段落
- 允许未整理完的资料先用摘录回退
- 先做少量高质量样板，再批量扩展

## 12. 结论

本次设计将参考资料页从“摘要 + 附件 + 回链”的资料卡，升级为“编者说明 + 可阅读正文 + 归档索引”的站内资料页。

它既保持了当前 Astro + GitHub Pages 的静态发布优势，也为后续的视频稿归档、资料互链、公开知识库建设提供了更接近 wiki 的阅读骨架。

第一阶段不追求自动化翻译和全量覆盖，而是先建立一个足够稳定、可持续维护、可逐步扩充的公开资料阅读模型。
