# 2026-07-08 全局壳层与照片墙设计

## 概述

本设计用于在现有 Astro 归档站上继续补齐一套更接近 `heiehiehi/XinghuisamaBlogs` 的全局体验层，同时保留当前已经成形的静态发布方案、内容结构和本地后台维护路径。

本轮不是重做内容系统，而是在已经上线的首页参考站风格基础上，继续完成以下几类缺口：

- 全局音乐运行时需要从“首页局部功能”升级为“跨页面连续播放”
- 非首页需要有持续可见的右下角悬浮播放器
- 首页搜索栏的材质与层级还不够接近参考站
- 站点需要补齐一次性入场加载动画与全局鼠标点击特效
- 站点需要新增独立照片墙，并提供本地后台入口管理相册与图片

本设计遵循用户已经确认的实现顺序：

1. 先做全局基础
2. 再做页面层视觉与交互
3. 最后把可维护入口补进本地后台

## 已确认目标

- 公开站点继续使用 Astro 静态构建并部署到 GitHub Pages。
- 前台整体气质继续以参考站为对标对象，尤其是首页、导航、搜索、播放器、加载动画和照片墙。
- 首页继续保留当前已经成形的参考站式主卡布局，不回退到普通博客首页。
- 音乐能力保留现有“云音乐 API + 歌曲 ID 配置”结构，不切换为纯本地音频列表。
- 音乐在页面切换时不中断，并在非首页有悬浮播放器继续控制。
- 新增独立照片墙页面，并以“相册总览 -> 相册内瀑布流 -> 大图查看”三层结构呈现。
- 新增照片墙的独立内容模型与本地后台编辑入口。
- 后台里与站点壳层相关的字段继续使用中文标签，不要求用户手改 JSON 才能维护。

## 非目标

- 本轮不把项目迁移为 Next.js 单体。
- 本轮不替换现有文稿、资料页、笔记页和专题页的核心内容模型。
- 本轮不引入外部 CMS 或托管后台。
- 本轮不改变 GitHub Pages 静态站发布主线。
- 本轮不把音乐数据持久化到服务端数据库。

## 当前代码基线

现有仓库已经具备以下基础，可直接承接本轮设计：

- `src/layouts/BaseLayout.astro` 已承担全站壳层职责，但目前尚未挂入全局音乐、全局入场动画与点击特效。
- `src/pages/index.astro` 已完成参考站风格的首页主卡布局，并挂载了首页搜索栏、首页音乐卡和歌词条。
- `src/components/music/store.ts` 已提供共享音乐状态，但运行时 `MusicRuntime` 仍然只挂在首页。
- `manager/src/pages/PageBuilder.tsx` 与 `manager/src/lib/homePageConfig.ts` 已提供首页配置结构化编辑基础。
- `src/content/config.ts` 当前包含 `articles`、`references`、`notes`、`topics`、`drafts`、`vault`，尚未有照片墙相册集合。

这意味着本轮不需要推翻现有实现，而是围绕这些现有文件继续扩展。

## 总体方案

### 技术路线

继续保持“Astro 页面层 + React islands 交互层 + Nanostores 全局状态 + 文件型内容源 + 本地 manager 编辑器”的架构。

与参考站对齐的方式采用“行为与观感对标”，而不是直接移植其 Next.js 客户端代码。原因如下：

- 当前站点已经有自己的公开内容结构和构建流程
- 直接复用参考站客户端实现会让 Astro 页面壳层变得难维护
- 用户真正需要的是观感、比例、交互方式和维护体验的接近，而不是技术栈一致

### 分层原则

本轮功能分成三层，避免再次出现样式、状态和内容维护互相缠绕：

- 全局壳层层：背景、导航、音乐运行时、悬浮播放器、加载动画、点击特效
- 页面交互层：首页搜索栏、照片墙总览、相册页、大图查看
- 内容维护层：首页配置、相册内容集合、后台结构化编辑、前台公开过滤

## 信息架构调整

### 顶部导航

现有主导航在“首页 / 文稿 / 资料库 / 笔记 / 专题 / 关于”的基础上新增：

- 照片墙

该栏目属于公开页面，与文稿和资料库平级。其目的是承接图像素材归档，不把图片硬塞进文稿或资料正文列表。

### 首页入口策略

首页不新增新的独立内容大卡来承载照片墙，避免破坏当前已经调整好的首屏比例。

照片墙入口放在两处：

- 顶部导航中的“照片墙”
- 首页搜索系统可搜到相册标题与图片说明

这样既不打乱首页布局，也让照片墙保持站内可发现性。

## 全局壳层设计

### 1. 全局音乐运行时

#### 目标

- 页面切换时音乐不中断
- 首页保持大播放器体验
- 非首页显示右下角悬浮播放器

#### 实现方式

- 将 `src/components/music/MusicRuntime.tsx` 从 `src/pages/index.astro` 提升到 `src/layouts/BaseLayout.astro`
- 继续复用 `src/components/music/store.ts` 作为共享状态源
- 音频元素在站点生命周期内只初始化一次，不因页面切换卸载
- 首页大卡与内页悬浮播放器都只消费同一份共享状态

#### 展现规则

- 首页：
  - 显示首页大播放器卡
  - 隐藏悬浮播放器，但不卸载组件
- 非首页：
  - 不显示首页大播放器
  - 右下角显示悬浮播放器

#### 样式要求

- 悬浮播放器比例尽量贴近参考站：左侧旋转唱片、中间歌名与歌词、右侧播放控制
- 信息区需要固定最小高度和稳定宽度，避免歌名两行时撑高整体卡片
- 首页大播放器继续保留更完整的封面、进度和控制区

### 2. 全局加载动画

#### 目标

- 首次进入站点时有参考站式欢迎层
- 同一会话内不重复完整打断式播放

#### 实现方式

- 在 `BaseLayout` 挂载客户端 `SplashScreen` 组件
- 通过 `sessionStorage` 记录本次浏览是否已经看过欢迎层
- 动画结束时给根节点打上已完成标记，让页面自然解锁显示

#### 视觉要求

- 视觉语言尽量对齐参考站：头像光环、站点名称、小号状态字、进度条
- 文案改为本站语境，不保留参考站原词
- 节奏保持短，不让用户等待过久

### 3. 全局点击特效

#### 目标

- 站点拥有统一的鼠标点击反馈
- 特效不干扰阅读与操作

#### 实现方式

- 在 `BaseLayout` 挂载画布型 `ClickEffect` 组件
- 监听全局点击事件，在点击点生成单次扩散涟漪

#### 视觉要求

- 使用参考站同类的冷色系涟漪
- 不使用夸张的爱心、烟花、文字粒子等与站点风格不符的特效
- 移动端允许自动弱化，优先保证性能和可操作性

### 4. 轻量页面过渡

本轮允许在全站加入轻微的页面淡入或区块入场动画，但只作为补充：

- 不做重型路由切场特效
- 不让正文阅读区因为动画而闪烁
- 重点只放在首页、照片墙和悬浮播放器等高感知区域

## 页面交互层设计

### 1. 首页搜索栏重构

#### 目标

- 搜索功能保持现有站内检索能力
- 外观、层次和细节更接近参考站

#### 保留能力

- 搜索文稿
- 搜索公开资料页
- 搜索公开笔记

#### 结构设计

- 搜索框本体采用大圆角、浅色磨砂底、左侧内嵌图标
- 输入时展开独立下拉面板
- 每条结果展示三层信息：
  - 标题
  - 摘要
  - 标签或栏目名

#### 文案要求

- 所有字段名、空结果提示和辅助文案全部使用中文
- 不保留无意义占位文本
- 语气保持站内归档风格，不套用参考站原文

#### 行为要求

- 输入即时过滤
- 命中内容高亮
- 点击条目后关闭结果面板并跳转
- 点击搜索框外关闭面板

### 2. 照片墙页面

#### 页面角色

照片墙是站内独立公开栏目，用于归档图片型素材，不代替文稿、资料库或附件列表。

#### 总体结构

照片墙采用三层浏览结构：

1. 相册总览页
2. 相册详情页
3. 大图查看层

#### 相册总览页

对齐参考站的核心观感：

- 顶部有标题与说明
- 右侧或上方有搜索框
- 主体不是普通卡片墙，而是“相册堆叠封面”式总览

每个相册卡片展示：

- 封面图
- 标题
- 简介
- 日期
- 图片数量

悬浮反馈包含：

- 堆叠层轻微展开
- 封面图放大
- 底部信息显现

#### 相册详情页

点击总览中的相册后进入相册页，展示：

- 返回入口
- 相册标题
- 简介
- 日期
- 图片总数
- 瀑布流图片列表

瀑布流优先用于图片数量和长短不一的素材，避免强行切成固定高度网格。

#### 大图查看层

点击任意图片后进入覆盖式大图层，支持：

- 原图预览
- 图片说明
- 点击遮罩关闭
- 右上角关闭按钮

本轮不强制加入键盘左右切换，但结构上应允许后续扩展。

### 3. 照片墙搜索

搜索对象包括：

- 相册标题
- 相册简介
- 图片说明
- 标签

搜索结果表现分两组：

- 匹配的相册
- 匹配的单张图片

这样可以兼顾“我记得相册主题”与“我只记得某张图说明”的两种查找方式。

### 4. 首页与内页播放器关系

首页播放器与悬浮播放器不是两套独立播放器，而是两个不同外壳：

- 首页播放器承担展示感和完整控制
- 悬浮播放器承担跨页面连续控制

两者共享以下状态：

- 曲目列表
- 当前曲目索引
- 播放状态
- 当前时间
- 总时长
- 当前歌词
- 错误状态

## 内容模型与后台设计

### 1. 新增 `albums` 内容集合

当前 `src/content/config.ts` 尚无照片墙集合，本轮新增：

- `albums`

它与 `articles`、`references`、`notes`、`topics` 平级，不作为某篇文章的子资源存在。

### 2. `albums` 元数据结构

`albums` schema 直接并入现有 `packages/content-core/src/schemas.ts`，和当前 `articleSchema`、`referenceSchema` 一起维护，避免再拆出新的单文件造成内容模型分散。

相册级字段：

- `title`：标题
- `date`：日期
- `summary`：简介
- `cover`：封面图
- `tags`：标签
- `draft`：草稿状态
- `visibility`：公开或隐藏
- `pinned`：是否置顶
- `relatedArticles`：关联文稿 slug 列表
- `relatedReferences`：关联资料页 slug 列表
- `photos`：图片列表

图片级字段：

- `url`：图片地址
- `caption`：图片说明
- `alt`：替代文本
- `credit`：来源或备注
- `relatedReferences`：关联资料条目
- `relatedArticles`：关联文稿

### 3. 公开与隐藏规则

照片墙需要支持“有些素材先存着，不公开”的维护方式，因此区分两层可见性：

- `draft: true`：草稿，不进入公开构建
- `visibility: hidden`：已归档但不公开展示，仅本地后台可见

只有满足以下条件时，相册才会进入公开照片墙：

- `draft` 为 `false`
- `visibility` 为 `public`

### 4. 与现有内容的关联方式

相册不是独立漂浮内容，它需要能与站内其他页面互链。

本轮先采用轻量显式关联：

- 相册 frontmatter 中记录 `relatedArticles`
- 相册 frontmatter 中记录 `relatedReferences`
- 文稿或资料页正文中允许手动链接到 `/photowall/<slug>/`

本轮不强制建设完整“图片反向链接图谱”，但 schema 和页面结构要为后续扩展保留字段。

### 5. 本地后台扩展

#### 首页配置

继续扩展现有 `PageBuilder` 的结构化首页编辑能力，使其更完整地覆盖站点壳层字段：

- 头像
- 公告文字
- 背景图
- 搜索框占位文案
- 云音乐歌曲 ID
- 云音乐 API 地址
- 备用封面
- 空闲歌词文案
- 社交按钮链接

后台里这些字段继续保持中文标签，不把维护工作退回到原始 JSON。

#### 照片墙编辑器

新增独立“照片墙”编辑入口，不和首页配置混在同一页。

最小可用编辑能力包括：

- 新建相册
- 修改标题、简介、日期、封面图、标签、公开状态
- 批量维护图片列表
- 调整图片顺序
- 给图片填写说明和备注
- 关联文稿与资料页

本轮优先保证结构化编辑可用，不强求复杂拖拽或所见即所得图像裁切。

### 6. 类型与接口调整

为承接 `albums`，需要同步扩展 manager 类型：

- `manager/src/types.ts` 的 `ContentKind` 增加 `albums`
- 内容列表、读取与保存接口同步支持 `albums`
- 前台内容查询辅助函数增加公开相册检索

## 数据流设计

### 音乐数据流

1. 站点壳层从 `siteShell.music` 读取歌曲 ID 与 API 配置
2. 全局 `MusicRuntime` 在客户端请求云音乐接口
3. 标准化后的曲目、歌词与错误状态进入 `musicState`
4. 首页大播放器、歌词条、悬浮播放器各自订阅共享状态
5. 页面跳转不重置状态，仅切换不同播放器外壳的可见性

### 照片墙数据流

1. 本地后台写入 `albums` 内容文件
2. Astro 构建时读取 `albums` 集合
3. 前台 `/photowall/` 页面生成公开相册总览
4. 页面内客户端组件负责搜索、切换相册和大图查看
5. 隐藏或草稿相册不进入公开构建索引

### 首页搜索数据流

1. 构建阶段收集文稿、公开资料页、公开笔记、公开相册
2. 运行时首页搜索组件消费预构建索引
3. 公开相册进入首页搜索结果，但结果条目只展示相册级信息，不直接展开单张图片结果
4. 相册内图片说明只用于照片墙页内搜索，不进入首页全站搜索，避免首页结果噪音过大

## 错误处理与降级策略

### 音乐接口失败

- 若云音乐接口失败，播放器不应导致页面报错
- 首页播放器与悬浮播放器显示统一的降级提示
- 歌词条回退到空闲文案
- 保留封面与壳层 UI 的完整性，不出现大面积布局塌陷

### 无照片墙内容

- 若没有任何公开相册，`/photowall/` 仍然输出完整页面
- 页面显示“照片墙整理中”式空状态，而不是 404

### 搜索无结果

- 首页搜索与照片墙搜索都使用明确中文空状态
- 空状态不挤压整体布局

### 配置缺失

- 首页配置中若头像、封面、背景图等字段缺失，需要有稳定 fallback
- manager 保存配置前应做基础校验，避免写入非法 JSON 或空白数组污染站点壳层

## 分阶段实施建议
